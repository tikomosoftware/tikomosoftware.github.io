# Prism

> プログラムに光を当て、隠れた構造（型・依存・**副作用・失敗**）を見せる小さな言語。
> AI時代に「契約を読む」流儀を学ぶための設計実験。**Python 3 だけ**で動く（依存ゼロ、ビルド不要）。

**初心者の最初の言語ではありません。** 効果・失敗・網羅性・能力・高階型といった本格的な型の考え方を扱います。
狙いは速さや簡潔さではなく、(1) 人間が読んで**考え方を鍛える**こと、(2) AIが書いたコードが
**「何に触れ・どう失敗しうるか」を署名から読み取れる**こと（※計算の正しさそのものの保証ではない）。

---

## 3秒で価値がわかる — 検査器が「隠れた嘘」を弾く

**1. 副作用の書き忘れ**
```prism
hello : ()  <-
  show!console "hello"
```
```
line 1: hello: performs effect !console but its signature declares no effects (pure)
```
署名は「純粋（`()`）」と言うのに、本体はコンソールに書いている。`: () !console` と宣言するまで通らない。

**2. 失敗の処理し忘れ**
```prism
main : Num  <-
  parseNum("oops")
```
```
line 1: main: can fail with ?BadNumber but its signature declares no failures
```
`parseNum` は失敗しうる（`?BadNumber`）。`try`/`match` で処理するか、署名に `?BadNumber` を宣言するまで通らない。

**3. OR型の網羅漏れ**
```prism
Shape : Circle{ radius: Num } or Square{ side: Num }

area(s: Shape) : Num  <-
  s match
    Circle{radius}  =>  radius * radius * 3
```
```
line 5: match on s: non-exhaustive match on Shape: missing Square -- every `or` case must be handled (or add `_`)
```
`Square` の処理が抜けている。`and`型が全フィールドを*与える*ことを要求する双対 ── `or`型は全ケースの*処理*を要求する。

> どれも多くの言語は黙って通します。Prism は**署名を契約として強制**し、ズレを**行番号つき**で指摘します。
> （上の出力は実際の `prism check` の出力そのままです。）

## Prism が保証すること / しないこと

| 保証する | 保証しない |
|---|---|
| 未宣言の副作用を検出 | アルゴリズムの正しさ |
| 未処理の失敗を検出 | 計算量の良さ |
| OR型の網羅漏れを検出 | セキュリティ安全性 |
| 型の不一致を検出 | 無限再帰しないこと |
| capability の未充足を検出 | 仕様解釈の正しさ |

さらに静的に弾くもの: **宣言レコードのフィールド欠落**（`Pt{x:3}`）、**`include` の名前衝突**（フラット
併合での同名再定義）、**効果の粒度リーク**（`!console` 宣言が `!io` を漏らす）。実行時も**隠れた型強制**
（`"ab" * 2` の黙った文字列反復など）を拒否する ── *契約に書けない抜け道を残さない*。

契約は **「何に触れ・どう失敗しうるか」の輪郭**を保証する。**中身が正しいかは別途検証が要る。**
だから Prism の効用は「読まずに信頼」ではなく、**レビューすべき範囲を劇的に絞れる**こと。

## 使ってみる

```sh
prism check  examples/broken.prism    # 静的検査（わざと誤った例 → 指摘される）
prism run    examples/calc.prism      # 実行（ビルド不要、Python 3 だけ）
prism reveal examples/infer.prism     # 署名なしコードの契約を推論して表示
prism serve                           # ブラウザの遊び場（同じエンジンを Pyodide で実行）
prism test  [core|gallery|all]        # 回帰テスト（core=言語のみ・速い／all=既定）
```

`prism` は `python cli.py` のラッパー（PATHに置けば使える）。詳しくは [GETTING_STARTED](GETTING_STARTED.md)。
ブラウザ版は**同じ `prism.py`/`check.py` を Pyodide で実行** —— 移植も別実装もなし。

## 4つの声（中心思想を1分で）

あらゆる計算は4つの問いに同時に答えている。Prism は各次元に専用の記号（声）を与え、**決して混ぜない**。

| 声 | 問い | 記号 | 順序 |
|---|---|---|---|
| **事実 (fact)** | それは何か（型・データ） | `:` `and` `or` | — |
| **変換 (flow)** | 何から導かれるか（依存） | `<-` `=>` `\|>` | **なし**（矢印が真実） |
| **作用 (effect)** | 世界の何を変えるか | `!world` | あり（だから印をつける） |
| **失敗 (failure)** | どう失敗しうるか | `?Error` `try` `match` | — |

関数の署名は、この全部を書いた **「嘘のつけない契約書」**:

```prism
f(x: A) : B  !world  ?Error  <-  ...
--      ^型   ^作用    ^失敗
```

AIが本体（`<-` の右）を書き、人間は契約（署名）を読む。

> 第5の穴 **高階種 `F[_]`**、能力 **`capability` / `given`**、**時間の声 `~>`** は*上級*機能です。
> 入口では効果・失敗・網羅の3本柱に集中し、これらは [REFERENCE](REFERENCE.md) で扱います。

## 何である / 何でない

- **宣言的／関数型寄り。手続き型ではない。** `<-` は順序のない依存で、行順に意味はない。順序が要るときだけ `~>`。
- **実用（プロダクション）言語ではない。** I/O はコンソールのみ、標準ライブラリ・モジュール・パッケージ管理なし。
  机上で本物に動く **「エンジンの精密模型」** —— 燃焼の仕組み（意味論）は本物だが、通勤する車（実務装備）ではない。
- **言語コアは2ファイル** —— `prism.py`（インタプリタ）＋ `check.py`（検査器）。それ以外はツール・ドキュメント・サンプル。
- **骨格は閉じている。** ロードマップは完走（パイプ `|>`・効果の粒度・**多重ディスパッチ**・衝突検出まで）。
  残りの地平線には**意図的に線を引いた**（本格的な名前空間・実 I/O 拡張 ── [NOTES](NOTES.md) 所見#22）。
  *機能を足し続けるより、足さない理由を書ける方が「完成」に近い。*

## 設計原則

- 隠れた構造は、隠さず見せる。見せきれない所は**ツール**（検査器・`reveal`）で顕示する。
- 4つの声を混ぜない。混ぜないことが、考え方の訓練になる。
- 書きやすさより、**正しさが安く確認できること**。
- 黙って捨てるな。検査器の沈黙は「OK」ではなく「見えなかった」かもしれない。
