## 検査器 —— Prism の本体

作って分かったことだが、**Prism の価値は新記法ではなく型システムにある。**
記法だけなら飾りで、`!` も `?` も「書けること」と「検査されること」は別物だった。
`check.py` がその本体を担う:

- 各式を **(型, 作用集合, 伝播する失敗集合)** に合成する。
- 関数本体の作用・失敗 ⊆ 署名の宣言、を検査。はみ出せばエラー。
- **行多相**:`!e` / `?g` を呼び出しごとに引数から具体化して伝播。
- **`:` の声にも歯**:戻り値型・引数型・リスト要素・match の腕・算術オペランドを
  単一化して照合する。型が食い違えばエラー。
- **match 網羅性**:`or` 型・List・可謬値・Num/Text のいずれを相手にしても、
  全ケース（または `_`）を要求する。`and` が全フィールドを要求することの双対。
- **能力制約**:`given T: Cap` を呼び出し点で具体型の `provides` により成就させる
  （未充足はエラー、ジェネリック呼び出し側へは再宣言で伝播）。`provides` の完全性・一意性も検査。
- **能力メソッド**:`show`/`fmap` の呼び出しを所属能力へ解決し、受け手の型が `provides` するか
  を確認、戻り値型を与える。各インスタンスのメソッド本体も署名（戻り値・作用・失敗）と照合。
- **高階種 `F[_]`**:`F[T]` を `List[Num]` に分解単一化（`F:=List, T:=Num`）し、
  `F[U]` を `List[U]` に復元する。容器の形を保ったまま中身を変える型が書ける。
- **時間 `~>`**:`~>` で並べる各手順が作用か失敗を持つことを要求する
  （純粋な手順はエラー）。「純粋な flow に順序なし」の双対 —— 順序の可視化と検査。
- **補間も見通す**:`"...{expr}..."` の中身も実コード。補間式の作用は伝播し、失敗は
  処理を要求する —— 作用・失敗が**文字列に隠れて契約をすり抜けることを許さない**。
- **エラーは場所を持つ**:AST がソース行を持ち、すべての指摘が `line N:` で位置を示す
  （宣言違反は宣言の行、式の型エラーは式の行、制約は呼び出し点）。

### reveal —— 検査器が「教えてくれる」

テーゼ「**コンパイラが推論し、ツールが顕わにする**」のもう半分。署名を書かない関数に対し、
`--reveal` は本体から契約（型・作用・失敗）を推論して表示する:

```sh
$ python check.py --reveal examples/infer.prism
  ask : Num !{console} ?{BadNumber}      [inferred]   -- try parseNum(read!console) から
  greet(name) : Unit !{console}          [inferred]   -- 補間内の show!console を見抜く
  add(a, b) : Num                        [inferred]
  risky(s) : Num ?{BadNumber}            [inferred]
```

これまでの検査器は「弾く」だけだった。reveal で初めて検査器が**隠れた契約を可視化して教える** ——
Prism の存在理由（隠れた構造を顕わにする）そのものを動かすツール。

> **設計判断:`Unknown` は「弱い型検査」ではなく「グラデュアル型付けの境界」。**
> 注釈が**書かれていない**箇所（署名なし関数・ラムダ引数）でのみ `?` が何にでも適合する。
> 一方、**既知の型どうしが衝突したら、もう黙らない。** これは Prism の哲学
> ——「コンパイラが推論し、ツールが顕わにする」「注釈なし=まだ契約されていない」——
> と AI時代のテーゼ（AIは注釈つきで書く）に整合する。

`broken.prism` は作用・失敗の3誤りを、`mistyped.prism` は型の5誤りを含む。
どちらも v0 ランタイムは**黙って実行してしまう**。検査器はすべて指摘する:

```
FAIL  examples/broken.prism  -- 3 problem(s):
  - line 11: greetAll: performs effect !console but its signature declares no effects (pure)
  - line 16: divide: can fail with ?DivByZero but its signature declares no failures
  - line 23: <flow>: unhandled failure ?{BadNumber} in operand -- discharge it with `try` or `match`

FAIL  examples/mistyped.prism  -- 5 problem(s):
  - line 7:  label: returns Text but its signature declares : Num
  - line 11: call to double: expected Num, got Text
  - line 14: <list>: list elements have differing types: Num vs Text
  - line 20: <match>: match arms have differing types: Text vs Num
  - line 23: <flow>: arithmetic needs Num, got Text

FAIL  examples/nonexhaustive.prism  -- 4 problem(s):
  - line 10: match on s: non-exhaustive match on Shape: missing Triangle -- every `or` case must be handled (or add `_`)
  - line 18: match on s: arm `Hexagon` is not a variant of Shape
  - line 23: match on xs: non-exhaustive match on List: missing [] (empty)
  - line 29: match on n: non-exhaustive match on Num: add a catch-all `_` -- Num has unbounded values
```

エラーは `line N:` で**ソース行を指す** —— 宣言違反は宣言の行、型エラーは式の行、呼び出し制約は呼び出し点。
「契約を読んで検査器に従う」ために、検査器の指摘は必ず場所を持つ。

## 作って初めて分かったこと（設計の穴）

紙の上では一貫して見えても、動かすと別物だった。主な発見:

1. **「`?E` 関数は ok/fail 値を返すのか、生の値を返すのか」が未決だった。**
   実装で「fail 以外はすべて暗黙の ok」と緩めて辻褄を合わせた。実装が暴いた本物の分岐点。
2. **静的型がないと、可謬関数と無謬関数が実行時に区別できない** —— `?` は飾りだった。
3. **作用も同じく飾り** —— 署名に `!console` を書かなくても誰も文句を言わなかった。
4. **行順非依存が遅延サンクでタダで実現** —— 実装が設計思想を体現した（嬉しい確認）。
5. **複数行署名がパーサにまるごと捨てられ、検査器が「偽の OK」を出していた。**
   > 教訓:**パースできなかったものを黙って捨てる検査器は、検査器がないより悪い。**
   > 緑のチェックが「全部OK」ではなく「見えた範囲はOK」を意味してしまい、
   > Prism の前提そのものを掘り崩す。
6. **型単一化を入れて分かった:「`Unknown` を捨てる」は全廃ではなく、再定義だった。**
   署名なし関数とラムダ引数を完全に型推論するのは大仕事で、全廃するとそれらが全部エラーになる。
   正しい答えは **`Unknown` を「弱い検査」から「グラデュアル型付けの境界」へ意味づけ直す**こと ——
   *書いていない*箇所でだけ何にでも適合し、*既知の型どうしの衝突*はもう見逃さない。
7. **検査器がランタイムより厳密になった瞬間があった**:`1 + "x"` の暗黙の文字列化。
   v0 ランタイムは黙って繋ぐが、検査器はこれを拒否する。隠れた強制変換は
   「隠れた構造を顕わに」に反するからだ。→ ランタイム側も将来 `1 + "x"` を拒否すべき、という宿題。
   （→ 所見#21 で回収。`*` の黙った文字列反復まで含めて締めた。）
8. **網羅性を入れて分かった:「型」と「ヴァリアント」は別の所有関係を要した。**
   `or` 型を網羅検査するには、まず**捨てていた型宣言を捕捉**し、さらに
   コンストラクタ `Circle{..}` が**自分の所属する `or` 型（Shape）の値になる**よう
   variant→所有型の逆引きを持つ必要があった。網羅性は List・可謬値・Num/Text にも
   **一様な catch-all 規則**として載った —— 4次元モデルがまた追加機構なしで吸収した。
9. **網羅性のデモ例が、無関係の既存バグを炙り出した。** 補間 `{...}` の中にレコード `{...}` を
   入れると `interpolate` が**最初の**閉じ波括弧を拾い式が途中で切れて落ちた（ネスト非対応）。
   → 補間を波括弧の深さ対応にする宿題が surface した（後に解決）。
10. **能力制約が「4つ目の掟」として既存の形に綺麗に嵌まった。** `given T: Cap` の強制は、
    失敗 `?` の「伝播して処理せねばならない」と**同型**だった。新しい検査の骨格は不要で、
    2集合の照合だけで済んだ。
11. **文法の壁が先にあった**:`capability`/`provides` はキーワード始まりで `parse_def` が
    パースすらできなかった。`given T: Ord` の旧スキップ処理も最初の `:` で止まり**事実上壊れていた**。
    文法を直してから初めて意味論を載せられた —— *パースできないものは契約に書けない*。
12. **高階種は「対称性が先に正解を教えた」例だった。** flow が高階の穴 `(T)->U` を持つなら
    fact も高階の穴 `F[_]` を持つはず —— という対称性が設計を導いた。既存の `unify`/`subst` に
    各15行ほど足しただけで、`given F: Functor` ともそのまま合成できた。
13. **時間の声で4次元がついに全部揃った。** `~>` の核心は順序付けそのものではなく、
    **順序の*正当性*を検査する**ことだった —— 並べる各手順は作用か失敗を持たねばならない。
    「純粋な flow に順序なし」の厳密な双対で、効果・失敗の追跡を**逆向き**に使う。
14. **能力メソッドの本体検査が、契約のテーゼを末端まで貫いた。** メソッド呼び出しを
    「能力の型変数を余分な穴とし `given _: Cap` 制約を付けたジェネリック関数」に変換すると、
    **既存の呼び出し検査がそのまま再利用でき**た。“偽 Functor”（`fmap <- xs`）が `List[T] ≠ List[U]`
    として弾かれるのは、契約を本体まで強制できた証拠。
15. **状態の時間変化は新構文を足さずに表現できた。** `~>` に「値のパイプ」の意味を持たせると
    声が濁る。代わりに **`or`型上の遷移関数 + 効果ブロックの再束縛**で状態機械を書ける。
16. **契約は文字列にも漏れていた。** `"{askNumber(x)}"` のような補間は作用も失敗も追跡しておらず
    `: Text`（純粋）で通っていた —— 緑のチェックが嘘をつく、所見#5 の再来。補間式を synth に
    通すだけで塞がった。*契約に書けない抜け道があってはならない*。
17. **「弾く検査器」から「教える検査器」へ。** 推論はほぼ無料だった —— synth が既に (型,作用,失敗) を
    返すので、署名なし関数ではそれがそのまま推論契約になる。`--reveal` は出力の向きを変えただけ。
18. **ドッグフーディングが、デモでは出なかったバグを一発で出した。** Prism で式評価器（calc）を
    書いた瞬間、複数行の `or` 型が**1行目しかパースされない**実バグが露見した。
    小さな本物のプログラム1本が、18個のマイクロ例より雄弁だった。
19. **「できない」を実測が暴き、最小の言語変更が解いた。** ライフゲーム・迷路ソルバ（純粋格子の
    fold として実装、Python参照と一致を検証）まではO(n)の `nth` でも約2秒で通った。だが**反応拡散
    （Gray-Scott）は本物のチューリング模様に150回以上の反復が要り**、近傍参照 × 反復で数十秒に膨れた。
    根因は **リストが内部では Python の配列なのに、言語が O(1) 添字アクセスを露出していない**こと。
    解は新型でも可変性でもなく、**読み出しだけの組み込み `at`/`len`**（不変のまま O(1)）。これで
    ライフ7倍・迷路4.5倍、反応拡散も実用域に。**判断軸は「純粋性を壊さない最小の追加」** ——
    格子シミュは毎ステップ全格子を作り直す純粋スタイルなので、必要なのは O(1) *読み出し*だけで、
    `set`/可変配列は不要だった。`floor` 追加と同じ「最小の変更で最大の能力」。教材用の `nth`/`length`
    （O(n) 構造的再帰）は残し、**抽象と機械コストの差**を見せる教育機会にした。
20. **「レコード構築の完全性」を入れる過程で、また"緑の嘘"が出た（所見#5の三度目）。** 宣言済み
    レコードを全フィールド揃えずに作る（`Pt{x:3}`）のは `and`（全フィールド在る）の双対として弾くべき。
    実装は Ctor synth で宣言名との差を取るだけ——だが**無署名の `picture <- [..]`（ValDef）は本体が
    そもそも synth されていなかった**ことが発覚し、検査が静止画デモに一切効かなかった。`d.sig` がある時
    だけ検査していた条件を外して**常に本体を synth**（署名照合だけ任意）に変えたら、全デモ無傷のまま
    完全性が効いた——*未検査の隙間は、検査がないのと同じ*。追加フィールドは開いたまま許可（グラフィクスの
    オプション `h/a/fill`）、未宣言コンストラクタ（ad-hoc な `Cell{u,v}`）は要件なし。
21. **所見#7 の宿題（ランタイムの隠れた強制変換）を回収。** 検査器は既に `1 + "x"` 等を弾くが、検査を
    迂回した時のランタイムが甘かった——`"ab" * 2` が Python の文字列反復で**黙って `"abab"`** を返し、
    `"a" - 1` / `"x" < 1` は**生の Python 例外文**を漏らしていた。算術 `- * /` を Num 必須に、順序比較は
    **同種同士**のみ（Text 対 Num を拒否）に統一し、Prism の言葉でエラーを出すようにした。*契約に書けない
    抜け道は、検査器が締めてもランタイムが開けていてはならない。*
22. **「完成形」は機能の数ではなく境界の明確さだった。** 残りの地平線を片付ける段で、二手に分かれた。
    *入れるべきもの*：パイプ `|>`（flow 声の純粋糖衣。脱糖して Call になるだけなので型・効果・失敗は素通り。
    `~>` を汚さない＝所見#15 を尊重）と、include の**名前衝突検出**（フラット併合での同名再定義を拒否。
    実際に `ulam-spiral` の定数 `maxN` が lib の `maxN` 関数を黙って隠していた地雷を発見・修正）。
    *引かないと決めた線*：効果の粒度（実 I/O が `!console` 一つしか無く、足場だけ作るのは無意味）、
    多重ディスパッチ（能力は1型変数の設計で、必要とする具体例が無い）、本格的名前空間（言語の性格を変える）。
    —— *機能を足し続けるより、足さない理由を書ける方が「完成」に近い。* 言語の骨格は閉じたまま。
23. **効果の粒度は「サブサンプション」で実体を得た。** 「`!io` を入れても中身が無い」と保留していたが、
    *粗い効果が細かい効果を一方向に包含する*関係（`!io ⊇ !console`）を入れれば、実効果が1つでも機構は
    本物になる：`!io` 宣言の本体は `!console` を使えて通り、`!console` 宣言は `!io` を行う本体を弾く
    （リークを許さない）。検査器の効果リーク判定に包含を一行足すだけで、関数本体にも能力メソッド本体にも
    同じ規則が載った。実効果（file/net/...）が増えれば即活きる前方互換。— 多重ディスパッチとの違い：
    こちらは*既存機構への一行の一般化*、あちらは*多引数能力への作り直し*。同じ「保留」でも費用が桁違いで、
    片方だけ進めたのはそのため。

24. **多重ディスパッチは「作り直し」ではなく「一般化」だった。** 当初は*多引数能力*をパーサ＋検査器＋
    ランタイムの大改造と見積もって保留したが、実際に着手すると**統一できる**と分かった：能力の型変数を
    *リスト*にし、各メソッドを「型変数で型付けされた引数位置でディスパッチ」「インスタンスは型タプルで登録」
    と定義すると、**単一ディスパッチは長さ1のタプルという特殊ケースに自然に吸収**された（`compare(a:T,b:T)`
    すら (Num,Num) で索かれ、`compare(3,"x")` が早期に弾かれる副産物つき）。ランタイムは args[0] 固定を
    「型変数位置のタプル」へ一般化するだけ、検査器は >1 変数の能力にだけ新パスを足し、既存の単一パスは不変。
    所見#22 で「用途が無いから保留」と書いたが、*やってみたら既存機構の一般化で、骨格をむしろ綺麗にした*。
    —— 見積もりの「8箇所の作り直し」は、正しい抽象（型変数リスト＋位置ディスパッチ）を見つけた瞬間に
    「数箇所の一般化」に縮んだ。**保留の理由が「費用」なら、まず正しい抽象を探すべきだった。**
    （後日、AI協働で**じゃんけん**`examples/rps.prism` を作り、勝敗を `Beats for A, B` の9インスタンスで
    決めた ── 「用途が無い」と言った機能に、直感的な生きた用途が出た。`Rock provides Beats`（型1つ）は
    arity 検査で弾かれ、多重ディスパッチが本物に検査されていることも確認できた。）

25. **AIコラボの実演（RPN電卓のドッグフード）が、また"緑の嘘"を一発で炙り出した（所見#5/#16 の四度目）。**
    「人間が契約＝署名を書き／AIが本体を書き／検査器が審判する」を実際にやった。RPN電卓は一発で `check` 緑・
    全ケース正常 ── だが「もし AI が `rescue` の一句を書き忘れたら?」を試すと、**`check` は緑なのに実行時に
    未捕捉の失敗でクラッシュ**した。原因は `EffectCall` の synth が引数の*作用*は拾うのに**失敗 (`ap`) を
    `set()` で握り潰していた**こと ── `show!console "{try solve(line)}"` の中で伝播する失敗が検査器から
    見えず、`main : () !console`（失敗ゼロ）が嘘をつけた。所見#16（補間の作用追跡）の兄弟バグ。`ap` を
    返すよう一行直すと、忘れた `rescue` 句が即座に検出された（全回帰テスト緑のまま＝既存デモは誰もこの穴に
    依存していなかった＝無害な潜在バグ）。**小さくても"本物のプログラム"1本が、数百のテストより雄弁**
    （所見#18 の再来）。付随して、`risky : Num ?Boom <- fail Boom` のような**nullary 値の宣言失敗が
    globals に保存されていない**小穴も見つけた（ret 型しか登録していない。値は稀なので記録のみ）。
    `words`（Text→単語列）も、入力行をトークン化する契約から必要が判明して足した最小の純粋組み込み。
26. **AIコラボ実演の3本目（自販機）が、レキサの実バグを炙り出した。** 時間の声 `~>` ＋ 状態機械の題材で
    自販機を作ったら、プロンプト文字列 `"… -- coin …"` の中の `--` が**行コメントとして剥がされ、
    後ろの `~> step(read!console)` ごと消えて**いた（だから1回表示で停止、しかも `check` は緑）。原因は
    レキサのコメント除去が `re.sub(r'--.*$','')` の**文字列無視の素朴版**だったこと（コメント自身が
    「v0: simple split」と白状していた）。文字列リテラル（と `\` エスケープ）を尊重して `--` を切るよう
    直し、`"a -- b"` がそのまま出るようになった（全回帰テスト緑＝既存は無依存）。**これでドッグフードが
    見つけた実バグは3本：calc＝複数行 or 型（#18）、RPN＝effect 引数の失敗漏れ（#25）、自販機＝文字列内 `--`。
    題材を変えるたびに、別の層の "緑の嘘" が出る** ── 本物のプログラムを書くこと自体が最良のテスト。

**結論:Prism の4つの声（事実・変換・作用・失敗）+時間が、すべて記法・評価器・検査器に揃い、**
**型システムは型単一化・行多相・網羅性・能力（制約/メソッド/本体）・高階種・時間・補間まで一周し、**
**`--reveal` で「推論して顕わにする」テーゼの後半も動いた。そのすべてが既存の仕組みの再利用だけで**
**載った —— 声は本当に直交していた。当初の「4つの次元」テーゼは、設計として閉じ、しかも自分自身で**
**書けるまでになった。**

> ロードマップは**完走**。「完成形」とは機能数ではなく**明確な境界**：以下は意図的な**非目標**として
> 線を引いた（所見#22）。
> - **効果の粒度（`!io` 等）** —— ✅ サブサンプションとして実装した（`!io` が `!console` 等を一方向に包含。
>   所見#23）。実効果はまだ `!console` のみだが、関係は実装済みで、効果が増えれば即活きる。
> - **能力メソッドの多重ディスパッチ** —— ✅ 実装した（`capability C for A, B` / `T provides C for U` /
>   実行時は複数引数の型でディスパッチ。所見#24）。単一ディスパッチは「型変数1個＝長さ1の組合せ」の
>   特殊ケースとして同じ機構に吸収された。
> - **`~>` の値パイプ糖衣** —— 所見#15 で却下済み（声が濁る）。代わりに別演算子 `|>` を入れて解決した。
> - **実用ランタイム（コンパイル / 名前空間 / モジュール）** —— `include` のフラット併合＋衝突検出で v0 は
>   足りる。修飾名つきの本格的名前空間は「Prism が何であるか」を変えるので v0 の外。
>
> いずれも言語の骨格は変えない —— **骨格は閉じており、育てるのはライブラリ**。
