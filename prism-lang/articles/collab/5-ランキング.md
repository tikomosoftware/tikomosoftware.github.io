# 5 — ランキング表（本格的な型システムを一度に使う）

コード: [`examples/leaderboard.prism`](../../examples/leaderboard.prism)

## これは何ができるか
プレイヤーをスコア順（降順）に並べて表示します。

```
python cli.py run examples/leaderboard.prism
→ 1. Cleo (150) / 2. Ada (120) / 3. Dan (95) / 4. Bob (90)
```

## 主役の機能：レコード＋ジェネリクス＋能力制約（型システムの本丸）
この回は、それまでの4本で**使っていなかった機能**を狙って一度に踏みました。Rust や Haskell の型システムに
似た、本格的な部分です：

- **レコード（`and`型）と `.field`**：`Player : Player{ name: Text, score: Num }` を作り、`p.score` で読む。
- **能力 `Ord`（比較できる、という性質）**：`Player provides Ord` でプレイヤーの比較方法（スコアで）を定義。
- **ジェネリクス `for T`**：`sortByOrd for T ...` ── *どんな型でも*ソートできる汎用の関数。
- **能力制約 `given T: Ord`**：「ソートできるのは*比較できる型だけ*」を型で要求し、呼び出し時に検査。

```
sortByOrd for T given T: Ord (xs: List[T]) : List[T]  <-  ...
```
これは「Tは何でもいいが、**Ordを備えた型に限る**」という意味。`Player` はOrdを備えているのでソートできる。

## 何が面白かったか ── 「未検証の組合せ」が正しく合成された
バグは出ませんでした。でも収穫があります ── これだけ高度な機能（レコード・ジェネリクス・能力・制約）を
**組み合わせても破綻しなかった**ことを確認できた。さらに型システムが**この新しい面でも効く**証拠も：

> 比較方法を持たない型（例：文字のリスト）をソートしようとすると、検査器が弾く ──
> 「**Text does not provide Ord**（Textは比較方法を持っていない）」。

「比較できない物は並べられない」を、実行する前に・型で止める。これが能力制約の力です。

## 正直なところ
ただのランキング表です。でも中身は、**現代の本格言語が持つ型機能**（trait/typeclass、ジェネリクス、制約）を
一通り使っています。RPN＝失敗、じゃんけん＝多重ディスパッチ、自販機＝時間と状態、数当て＝ループ、に対して、
これは「**型で安全に抽象化する**」という型システムの本丸を見せる一本です。これで5本が言語をほぼ一周しました。
