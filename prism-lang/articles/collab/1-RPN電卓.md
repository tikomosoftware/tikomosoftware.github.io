# 1 — RPN電卓（そして検査器のバグを見つけた本）

コード: [`examples/rpn.prism`](../../examples/rpn.prism)

## これは何ができるか
逆ポーランド記法（演算子を後ろに書く方式）の電卓です。1行入力して計算します。

```
echo "3 4 + 5 *" | python cli.py run examples/rpn.prism
→ = 35              （(3+4)*5 = 35）
```
そして**ありえる失敗を全部きちんと扱います**：
- `3 0 /` → 「division by zero」（ゼロ除算）
- `1 2 x` → 「unknown token」（知らない記号）
- `1 +` → 「not enough operands」（数が足りない）
- `1 2 3` → 「too many operands left」（数が余った）
- 空入力 → 「empty expression」

## 主役の機能：失敗の声 `?`
ふつうの言語では「この関数、どんな失敗をしうるの?」は中身を読まないと分かりません。Prism は**起きうる失敗を
署名に並べます**：

```
solve(line: Text) : Num  ?BadToken ?DivByZero ?Underflow ?Empty ?Leftover
```

この一行が「solveは数を返す、ただし**この5通りで失敗しうる**」と宣言しています。そして検査器が
「その5通りを全部 `main` で処理したか?」を強制する。処理し忘れたらコンパイルが通りません。

## 何が面白かったか ── 作ったら、検査器の本物のバグが出た
完成後、「もし私（AI）が失敗処理の一句を**書き忘れたら**?」を試しました。すると ──

> **検査器は「OK（緑）」と言うのに、実行すると未処理の失敗でクラッシュした。**

これは Prism にとって最悪の事態（「緑なのに嘘」）です。原因を追うと、検査器が
`show!console "{ ... }"` の**中で起きる失敗を見落としていた**（表示の作用は追うのに、失敗だけ捨てていた）。
**一行直して**塞ぎ、忘れた処理がちゃんと検出されるようになりました。

机上のテストでは見つからなかった穴が、**実際の小さなプログラム1本**で出た ── これが「作ってみる」価値です。
（おまけ：入力行を単語に割る `words` という小さな道具も、この契約から必要だと分かって足しました。）

## 正直なところ
ただの電卓です。でも「**失敗を握りつぶせない電卓**」── どう失敗しうるかが署名に出ていて、検査器がそれを守る。
そこが普通の電卓と違います。
