## TL;DR

- **Prism** は「AI時代に*契約を読む*訓練」のための、ペダゴジー優先のミニ言語（設計実験）です。
- 関数の**署名に、型・副作用・失敗・順序が全部出る**。静的検査器がそれを強制します。
- **Python 3 だけ**で動く（依存ゼロ・ビルド不要）。言語コアは `prism.py` ＋ `check.py` の **約1,900行**。
- 同じエンジンが **ブラウザ（Pyodide）でも**動き、約**300枚**のビジュアル・ギャラリー付き。
- **実用言語ではありません。** 「AIが書いたコードの、人間が読むべき範囲を絞る」考え方を体験する教材です。

## きっかけ：AIが本体を書くなら、人間は何を読む?

AI にコードを書かせるのが当たり前になりました。でも「動いてそう」を信じるには、結局**全行レビュー**します。
レビューを減らしたい。では **何を読めば、読まない部分の振る舞いを絞れるのか?**

普通の言語だと、関数の見た目（型シグネチャ）からは「副作用があるか」「どう失敗しうるか」「順序が要るか」が
**読み取れません**。`def process(x):` は、ログを吐くのか、例外を投げるのか、ネットを叩くのか ── 署名は黙っています。

Prism の賭けは単純です。

> **それらを全部、署名に出して、検査器に強制させる。**
> そうすれば、署名が「何に触れ・どう失敗しうるか」の*外枠*を保証し、レビューを中身（正しさ）に集中できる。

正しさそのものは保証しません。でも「読むべき範囲」は劇的に狭まります。

## 30秒デモ：検査器が"隠れた嘘"を弾く

百の説明より、3つの「怒られ方」を見てください。どれも多くの言語は黙って通します。

**1. 副作用の書き忘れ**

```prism
hello : ()  <-
  show!console "hello"
```
```
line 1: hello: performs effect !console but its signature declares no effects (pure)
```
署名は「純粋（`()`）」と言うのに、本体はコンソールに書いている。`: () !console` と宣言するまで通りません。

**2. 失敗の処理し忘れ**

```prism
main : Num  <-
  parseNum("oops")
```
```
line 1: main: can fail with ?BadNumber but its signature declares no failures
```
`parseNum` は失敗しうる（`?BadNumber`）。`try`/`match` で処理するか、署名に書くまで通りません。

**3. OR型の網羅漏れ**

```prism
Shape : Circle{ radius: Num } or Square{ side: Num }

area(s: Shape) : Num  <-
  s match
    Circle{radius}  =>  radius * radius * 3
```
```
line 5: match on s: non-exhaustive match on Shape: missing Square -- every `or` case must be handled (or add `_`)
```
`Square` の処理が抜けている。網羅するまで通りません。

この3つだけで、Prism が何を嬉しがっている言語か、だいたい伝わると思います。

## 中心の考え方：計算の「4つの基本の声」＋時間

Prism は、あらゆる計算が同時に答えている問いを4つに分け、**専用の記号を与えて決して混ぜません**。

| 声 | 問い | 記号 |
|---|---|---|
| **事実 (fact)** | それは何か（型・データ） | `:` `and` `or` |
| **変換 (flow)** | 何から導かれるか（依存） | `<-` `=>` `\|>` |
| **作用 (effect)** | 世界の何を変えるか | `!world` |
| **失敗 (failure)** | どう失敗しうるか | `?Error` `try` `match` |

そして**5本目＝上級の声**として、順序そのものを扱う **時間 (time) `~>`** があります（「これ、その次にあれ」が
要るときだけ。純粋な `<-` には順序がありません）。

関数の署名は、この全部を一か所に書いた**嘘のつけない契約書**になります。

```prism
divide(a: Num, b: Num) : Num  ?DivByZero  <-  ...
--                       ^型    ^失敗        ^本体
```

そして検査器（`check.py`）が**本体が契約どおりか**を強制する。**Prism の価値は新しい記法ではなく、この検査器**に
あります ── `!` も `?` も「書けること」と「検査されること」は別物だからです。

## もう一歩：型システムは一周している

教材とはいえ、型まわりはちゃんと一周しています（すべて検査される）。

- **代数的データ型と網羅性**（`and`＝全フィールドを与えよ／`or`＝全ケースを処理せよ、という双対）
- **効果と失敗の行多相**：1つの `map` が、純粋・副作用つき・可謬な関数すべてに対して*正直*なまま使える
- **能力（capability / given）＝ trait/typeclass**、**高階種 `F[_]`**（Functor など）
- **多重ディスパッチ**：`capability Collide for A, B` のように複数の型変数を持て、`hit(ship, asteroid)` が
  **両引数の型**で実装を選ぶ（単一ディスパッチは「長さ1の組合せ」として同じ機構に吸収）
- **グラデュアル型**：署名を省いた所だけ `_` で緩み、*既知の型どうしの衝突*はちゃんと弾く
- **パイプ `|>`**：`x |> f` は `f(x)` の純粋な糖衣（時間の声 `~>` はあえて汚さない）
- **`--reveal`**：署名なしの関数の契約（型・効果・失敗）を**推論して見せる**

そして「書けない抜け道を残さない」── 文字列補間 `"{askUser()}"` の作用も、`"ab" * 2` のような隠れた型強制も、
検査器とランタイムの両方で塞いであります。

## 証明としてのギャラリー（でも本質は検査器）

`<-` は順序のない依存、図形は純粋な値 ── という制約のまま、約**300枚**のデモが描けます。マンデルブロや
ストレンジアトラクタ、ライフゲーム、反応拡散（チューリング模様）、ゲームのエフェクトまで。

（▶ ここにアニメ GIF：ローレンツ・アトラクタ／斬撃エフェクト／反応拡散 など2〜3枚）

ただし強調したいのは ── **これはグラフィックDSLではありません**。ギャラリーは「**小さな言語＋少しの
ライブラリで、検査器を通したままここまで行ける**」という*言語の証明*です。たとえば「状態を持つ計算」も、
**不変グリッドへの fold** として純粋に書けます（ライフゲームは Python 参照と完全一致を検証済み）。

すべてブラウザの遊び場で動きます ── **移植や別実装ではなく、同じ `prism.py`/`check.py` を Pyodide で
実行**しているだけです。

## 何でないか（先に言っておく）

期待値を間違えないために、はっきり書きます。

- ❌ **実用（プロダクション）言語ではない**：I/O はコンソールのみ。モジュール・パッケージ管理・実I/Oなし。
- ❌ **初心者の最初の言語ではない**：効果・失敗・網羅・能力・高階型を扱う、概念的に濃い言語。
- ❌ **「AIのコードを読まずに信頼できる」言語ではない**：保証するのは*振る舞いの外枠*で、計算の正しさは別。
- ❌ **安全性を完全保証する言語ではない**。

机上で本物に動く**「エンジンの精密模型」** ── 燃焼の仕組み（意味論）は本物だが、通勤する車（実務装備）では
ない、という位置づけです。これは欠陥ではなく狙いです。

| Prism が保証すること | 保証しないこと |
|---|---|
| 未宣言の副作用を検出 | アルゴリズムの正しさ |
| 未処理の失敗を検出 | 計算量の良さ |
| OR型の網羅漏れを検出 | セキュリティ安全性 |
| 型の不一致を検出 | 無限再帰しないこと |
| capability の未充足を検出 | 仕様解釈の正しさ |

## 実装メモ（軽く）

- **言語コアは2ファイル**：`prism.py`（レキサ＋パーサ＋ツリーウォーク評価器）＋ `check.py`（静的検査器）。合計 **約1,900行**。
- 小さいのは、(1) ツリーウォークで最も単純、(2) 数値・文字列・メモリは Python におんぶ、(3) 射程を意図的に絞った、
  (4) 4つの声が直交していて新機能が既存機構（`unify`/`subst`）に載るから。
- **依存ゼロ・ビルド不要**。回帰テストは `core`（言語のみ・数秒）／`gallery`／`all`（約800アサーション）に分割。
- ブラウザ版は Pyodide（WASM版CPython）で**同じコア**を実行。

開発の進め方そのものが、テーゼの実演になっています ── **人間が署名（契約）を書き、AIが本体を書き、検査器が
審判する**。実際、Prism で式評価器を Prism 自身に書かせたら、複数行 `or` 型のパース漏れという実バグが一発で
露見しました（小さな本物のプログラム1本が、マイクロ例より雄弁でした）。

## まとめ

研究的に新しい概念は入っていません（effect system も Result も typeclass も既存）。Prism の独自性は、それらを
**AI時代に「契約を読む訓練」として体験できる、小さな処理系・小さな検査器・小さなCLIにまとめた**ことにあります。

> AIにコードを書かせる時代に、人間が読むべき「契約」とは何か。

このテーマを、純粋で小さな世界で手を動かして掴める ── 自作言語・静的検査器・AI時代の契約駆動という切り口の、
教材であり設計実験です。ここで掴んだ感覚を、実務の言語に持ち帰ってもらえたら本望です。
