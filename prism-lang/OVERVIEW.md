## 1. これは何か / 何でないか

**Prism は、AI時代に「学ぶ価値」のあるペダゴジー優先のプログラミング言語**（設計実験）です。
狙いは実務の速さや簡潔さではなく、次の二つ:

- **人間が読んで「考え方」を鍛えられること**
- **AIが書いたコードが「何に触れ・どう失敗しうるか」を契約（署名）から読み取れること**
  —— **全行を同じ密度で読むのではなく、レビューを絞れる**（※契約は効果と失敗を保証するが、計算の正しさそのものは別途検証が要る）

> **初心者の最初の言語ではない。** 効果・失敗・能力・高階型といった本格的な型の考え方を扱う、
> 「型や契約を学びたい人」向け。ビジュアルの遊び場は誰でも触れるが、言語そのものは概念的に濃い。

**これではない、を先に明確に:**

- **手続き型ではない。** `<-`（導出）は順序のない宣言で、行順に意味はない（遅延評価）。
  「これ、その次にあれ」と順序が要るときだけ `~>`（時間の声）で明示する宣言的／関数型寄りの言語。
- **実用（プロダクション）言語ではない。** I/O はコンソールのみ、標準ライブラリ・モジュール・
  ファイル/ネットワーク・パッケージ管理が無い。**机上で本物に動く「エンジンの精密模型」** ──
  燃焼の仕組み（意味論）は本物だが、通勤する車（実務装備）ではない。これは欠陥ではなく狙い。

## 2. 中心の考え方 ── 計算の「4つの基本の声」＋時間

あらゆる計算は、まず4つの問いに同時に答えている。Prism は各次元に専用の記号を与え、**決して混ぜない**。

| 声 | 問い | 記号 |
|---|---|---|
| **事実 (fact)** | それは何か（型・データ） | `:`  `and`  `or` |
| **変換 (flow)** | 何から導かれるか（依存） | `<-`  `=>`  `\|>` |
| **作用 (effect)** | 世界の何を変えるか | `!world` |
| **失敗 (failure)** | どう失敗しうるか | `?Error`  `try`  `match` |

そして**5本目＝上級の声**として、順序そのものを扱う **時間 (time) `~>`** がある（「これ、その次にあれ」が要るときだけ使う。
純粋な flow には順序が無い）。最初に学ぶのは上の4本、`~>` は発展機能 ── という整理で README / REFERENCE と統一している。

関数の署名は、この全部を一か所に書いた **「嘘のつけない契約書」** になる:

```prism
divide(a: Num, b: Num) : Num  ?DivByZero  <-  ...
--                       ^型    ^失敗        ^本体
```

そして検査器（`check.py`）が **本体が契約どおりか** を強制する。**Prism の価値は新しい記法ではなく、
この検査器（型システム）にある** ── 記法だけなら飾りで、`!` も `?` も「書けること」と「検査される
こと」は別物だから。

## 3. 他言語と何が違うか（特徴的な記法）

最も近い親戚は Rust（`Result`/trait）と Haskell（純粋性/型クラス）。Prism 固有の点:

- **契約に基本の声が出る** ── 型・作用・失敗が署名に揃い、必要なときだけ時間 `~>` も明示される（散らばらない）。
- **`<-` は順序のない導出** ── 上下を入れ替えても同じ。命令の列ではなく依存の宣言。
- **エラーは「型に書く OR 型」** ── `?DivByZero` は型の一部。`try`(伝播) と `match`/`rescue`(消費)
  の2通りだけで、隠れた例外経路が無い。
- **副作用が型に出る** ── `!console` が署名に強制され伝播。純粋＝ `!` が無いこと。
- **時間 `~>` を一級の検査対象に** ── 「順序付けする各手順は効果か失敗を持て。純粋な手順に
  順序は無い」を検査する。**「あなたは順序を必要としていない」と指摘してくれる言語**は珍しい。
- **ジェネリクス＝次元に空けた穴** ── `for T, U, !e, ?g, F[_]`。穴がどの次元かが記号で分かり、
  1個の `map` が純粋・副作用つき・可謬すべてを覆う。
- **AND/OR を対称に** ── `and`型（全フィールドを*与えよ*）と `or`型（全ケースを*処理せよ*＝網羅）
  を双対として教える。レコードは**構築も完全性を要求**（宣言フィールドを欠けば弾く）。
- **パイプ `|>` は flow の純粋糖衣** ── `x |> f` ＝ `f(x)`、`x |> f(a)` ＝ `f(x, a)`（データ先頭）。
  脱糖して呼び出しになるだけなので型・効果・失敗は素通り。時間の声 `~>` は**あえて**汚さない。
- **効果は粒度を持てる** ── `!io` が `!console` 等を一方向に包含。粗い効果で宣言しても細かい効果を
  本体で使えるが、逆（細かい宣言で粗い本体）はリークとして弾く。

```prism
-- 失敗・効果・網羅・能力が、すべて署名と検査に現れる例
Shape : Circle{ radius: Num }  or  Square{ side: Num }

area(s: Shape) : Num  <-          -- ? も ! も無い = 純粋・無謬
  s match                          -- Circle と Square を両方扱わないと検査器が弾く
    Circle{radius}  =>  radius * radius * 3
    Square{side}    =>  side * side
```

## 4. 実際の使い方（CLI / ブラウザ）

ビルド不要。**Python 3 だけ**で動く。統一CLIは `cli.py`（`prism` ランチャ同梱）。

```sh
prism run    examples/calc.prism      # 実行（= python cli.py run ...）
prism check  examples/calc.prism      # 静的検査（OK / FAIL を行番号つきで）
prism reveal examples/infer.prism     # 署名なしコードの契約を推論して表示
prism test [core|gallery|all]         # 回帰テスト（core=言語のみ・数秒／all=全部・既定）
prism serve                           # ブラウザの遊び場を起動
```

- **検査器は弾くだけでなく「教える」**。エラーは `line N:` で場所を示す。
- **reveal** は「コンパイラが推論し、ツールが顕わにする」の後半 ── 注釈の無い関数の契約
  （型・効果・失敗）を本体から推論して見せる。
- **ブラウザ版**（`playground.html`）は **同じ `prism.py`/`check.py` を Pyodide（WASM版CPython）で
  実行** ── 移植も別実装もなし。完全オフラインにするなら `prism fetch-pyodide`（ローカル優先）。

## 5. AI との組み合わせ（本来のテーゼ）

「AI連携」には二つの意味がある。混同しないこと。

**(A) 方法論としての連携 ── これが本命。** 特別なAPIは不要で、開発の流儀のこと:

```
人間   →  署名(契約)を書く
AI     →  本体(<- の右)を書く
検査器 →  本体が契約どおりか保証する（prism check → OK）
人間   →  署名で振る舞いの外枠を確認し、レビューを契約違反・境界値・アルゴリズムに絞る
```

ふつうは「AIのコードを信じるには中身を全部レビュー」。Prism の賭けは
**「型・効果・失敗・時間を署名に強制し検査器が一致を保証するなら、署名が*何に触れ・どう失敗しうるか*の
外枠を保証し、全行レビューではなく必要な部分（中身の正しさ）に集中できる」**（正しさそのものの保証ではない）。
連携の正体は **人間=契約 / AI=実装 / 検査器=審判** という分業。

**(B) 実際にLLMにPrismを書かせる手順。** AIはPrismを知らない（新言語）ので:

1. `REFERENCE.md` ＋ 例を数本、コンテキスト（システムプロンプト）に渡す。
2. **署名を先に書かせて** 本体を生成させる（＝(A)の分業）。
3. **`prism check` で検証 → エラーをそのままAIに返す → 直させる** を回す（エージェント・ループ）。

ここで効くのが検査器の質: エラーが**行番号つき・契約に焦点**なので、**そのままコピペで
AIに返せる**。`reveal` で推論契約を見せることもできる。

> 現実的な射程: Prism は実用言語ではないので「AIと実アプリ開発」ではなく、
> **小さく契約の濃いプログラムをAIと作り、「契約を読んで信頼する」流儀を体験・訓練し、
> その方法論を実務の言語に持ち帰る** ── という使い方になる。

## 6. 言語本体のファイルと全体構成

**言語の本体（エンジン）は2ファイルだけ:**

| ファイル | 役割 |
|---|---|
| **`prism.py`** | 言語そのもの ── レキサ＋パーサ＋ツリーウォーク評価器（`run` の中身） |
| **`check.py`** | 静的検査器 ── 型/効果/失敗/網羅/能力/高階種/時間 ＋ `--reveal` 推論 |

合計 約1,900行。小さいのは、(1) ツリーウォーク型で最も単純な実行方式、(2) 数値・文字列・メモリ・
例外など低レベルは Python に全部おんぶ、(3) 射程を意図的に絞った、(4) 4つの声が直交していて
新機能が既存機構(`unify`/`subst`)に載るから ── が理由。**この小ささ（意味論が一度に読み切れる）
こそ「学ぶ価値のある言語」という狙いに合致する。**

**骨格は閉じている。** ロードマップは完走（パイプ `|>`・効果の粒度・**多重ディスパッチ**・include 衝突
検出まで実装）。残りの地平線には**意図的に線を引いた**（[NOTES.md](NOTES.md) 所見#22）── 本格的な
名前空間（フラット併合＋衝突検出で足りる）、実 I/O 拡張（効果の関係は実装済み、効果自体は console のみ）。
*機能を足し続けるより、足さない理由を書ける方が「完成」に近い。*

**周辺（本体を呼ぶ道具・配布物）:**

```
cli.py  prism  prism.cmd          統一CLIとランチャ
test.py                           回帰テスト（core / gallery / all に分割）
playground.html                   ブラウザの遊び場（中で本体を動かす／300枚のギャラリー）
lib/(12)                          include 可能な標準ライブラリ（list/math/picture/…）
examples/(34)  tutorial/(6)  algorithms/(9)   サンプルコード
README / GETTING_STARTED / TUTORIAL /          ドキュメント
  REFERENCE / ALGORITHMS / NOTES / IDEAS / OVERVIEW.md
LICENSE  .gitignore
pyodide/                          (任意) fetch-pyodide で取得、未コミット
```
